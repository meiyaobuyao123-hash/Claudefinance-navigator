# 明理 Agent v2 — 整体架构设计

> 版本：v1.0 | 最后更新：2026-03-22 | 作者：产品团队

---

## 1. 背景与目标

### 现状（v1 痛点）

| 痛点 | 影响 |
|------|------|
| 每次对话重置，无用户记忆 | 用户要反复描述自己的情况，体验差 |
| 不知道用户真实持仓 | 建议泛泛，无针对性 |
| 等待 3-8 秒才出字 | 用户以为 App 崩了 |
| Markdown 显示原始符号 | 内容再好也显得粗糙 |
| 没有护栏，依赖 prompt 防止越界 | 可被 prompt injection 绕过 |
| 没有任何评估数据 | 不知道明理在哪里表现差 |

### v2 目标

```
让明理从"每次重新认识你的 AI"变成"真正了解你资产情况的私人顾问"
```

具体指标：
- 首次对话即可给出个性化建议（冷启动 < 30 秒）
- 明理"知道"用户的真实持仓，建议可落地到具体持仓
- 响应首字符延迟 < 500ms（流式）
- Prompt injection 拦截率 100%（确定性规则）
- 有可量化的用户满意度数据

## 3. 模块优先级与依赖关系

```
P0（基础，必须先做）
  ├── M01 冷启动引导         → 解决"新用户第一次体验"
  ├── M02 持仓上下文注入     → 解决"AI不认识你持仓"
  └── M03 分层Prompt架构    → 解决"prompt膨胀"

P1（核心升级）
  ├── M04 对话状态机         → 依赖 M03
  ├── M05 Tool Use          → 依赖 M03
  ├── M06 流式输出           → 独立，可与P0并行
  └── M07 护栏机制           → 独立，可与P0并行

P2（长期质量）
  └── M08 评估与反馈         → 依赖所有P0/P1完成
```

---

## 4. 关键设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 记忆方式 | System Prompt 注入 | 用户档案小而结构化，RAG 过重 |
| 工具触发 | 混合（规则+AI） | 可预测场景用规则，复杂场景用AI |
| Agent 模式 | 单 Agent + Tool Use | 金融场景需要可审计，多Agent复杂度高 |
| 摘要触发 | 超过 20 轮或 8000 token | 避免"context rot" |
| 护栏实现 | 客户端确定性规则 | 不依赖AI，100% 可测试 |
| 流式库 | anthropic_sdk_dart | Dio 不支持 SSE 流式 |

---

## 5. 非目标

- 不做多 Agent 编排
- 不做 RAG 知识库检索（现阶段）
- 不做语音输入/输出
- 不做图表生成
- 不自动执行任何交易操作
